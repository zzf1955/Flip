## 背景

当前项目已在 T032 中初步收敛训练主线、删除/归档 FunControl 与 RectFlow 等废弃入口，但仍存在以下问题：路径、数据、模型、输出、实验参数散落在脚本参数和文档中；Seedance、Human Mesh + inpaint、Mitty 训练等 pipeline 边界没有统一写入 Agent 入口文档；实验命名依赖人工传参，难以稳定记录每次实验使用的数据规模和训练类型；训练 Step5 文档没有按实际实验类型拆分；部分历史代码/文档仍容易让新实验误以为需要外部 DiffSynth 同步。

用户要求按 `/develop` 中大型任务流程整体重构：先形成任务计划，再在独立 worktree/分支中执行，不在 main 上直接做大改。

## 目标

1. 将路径管理、实验设置、默认输出位置、cache/log/checkpoint 约定统一收敛到 config 层。
2. 明确当前正式 pipeline，并写入 Agent 入口文档，便于后续直接根据实验类型执行。
3. 建立统一实验命名与参数管理：无需手动传实验名，由配置和 CLI 参数自动构造。
4. 每次实验自动记录数据用量、数据 split、关键参数、产物路径和训练/生成类型。
5. 训练实验类型通过参数指定，并覆盖 Mitty 直接外观 transfer 与 Mitty 三阶段 LoRA。
6. Mitty 三阶段训练自动检测 LoRA 维度，避免手动传错 rank/dim。
7. 彻底与外部 DiffSynth sync 解耦：新主线不依赖外部仓库同步、路径或隐式 fallback。
8. 重新整理文档，尤其把 Step5 train 拆成清晰子文档与执行手册。

## 范围

**包含：**

- 新增或重构 `src/core/config.py` 及相关 config 模块，统一项目根目录、数据目录、cache、log、checkpoint、tmp、Seedance 输出、Human Mesh/inpaint 中间产物等路径。
- 梳理 pipeline 类型枚举/配置，至少覆盖：

  - `seedance_human_mesh`：Human Mesh 渲染视频 + robot mask/inpaint 背景后调用 Seedance API 生成视频。
  - `train_mitty_transfer`：Mitty 直接外观 transfer。
  - `train_mitty_three_stage_lora`：Mitty 三阶段训练，三个 LoRA。

- 实验记录：生成结构化 manifest，例如 `training_data/experiments/<name>/manifest.json` 或统一 config 指定路径，记录数据量、样本列表/索引、参数、git 信息、命令、产物路径。

## Pipeline 设计草案

### A. Seedance API 直接生成视频

- 类型：`seedance_direct`。
- 输入：prompt / 参考图或参考视频 / 可选机器人原视频元数据。
- 输出：`output/seedance/<experiment_name>/` 或 config 中定义的生成输出目录。
- 记录：API 参数、输入文件、生成视频路径、帧率/分辨率/时长、数据条数。
- 正式入口：统一 CLI 参数 `--experiment-type seedance_direct`，避免散落脚本参数。

### B. Human Mesh + Inpaint 后 Seedance 生成视频

- 类型：`seedance_human_mesh`。
- 输入：G1 视频与 pose、SAM2/FK mask、背景 inpaint、Human Mesh/SMPLH 渲染视频。
- 数据流：robot mask → 背景 inpaint → Human Mesh render → Seedance API 生成/重绘。
- 输出：按 experiment name 分目录保存 mask、inpaint 背景、mesh render、Seedance 结果与 manifest。
- 要求：中间产物路径全部由 config 构造，不在脚本内硬编码。

### C. Train：Mitty 直接外观 Transfer

- 类型：`train_mitty_transfer`。
- 输入：`(human, robot)` paired video 数据、cache、可选 eval split。
- 输出：checkpoint、log、eval samples、manifest。
- 记录：训练数据量、eval 数据量、cache 命中情况、关键超参、模型基础版本。
- 要求：实验名自动生成，不再手动传入。

### D. Train：Mitty 三阶段 LoRA

- 类型：`train_mitty_three_stage_lora`。
- 阶段：三个 LoRA 分阶段训练/加载，阶段边界与产物写入 manifest。
- LoRA 维度：自动从 LoRA 权重或配置读取；多来源冲突或无法推断时直接报错。
- 输出：每阶段 checkpoint/log/eval、最终组合说明、manifest。
- 要求：CLI 只指定实验类型和必要数据/训练参数，不要求人工维护三个 LoRA dim。

## 配置与实验命名方案

### 配置层

- `src/core/config.py` 保留基础路径和环境常量。
- 可新增 `src/core/experiment_config.py` 或等价模块，负责实验类型、命名、manifest、路径构造。
- 所有 pipeline 入口通过 config API 获取路径，不直接拼接项目外路径。
- 允许通过环境变量覆盖 cache 根目录，但必须在 manifest 记录最终解析值。

### 命名规则

建议格式：

```text
YYYYMMDD-HHMMSS_<experiment_type>_<dataset_or_scene>_<split>_n<samples>_<keyparams>_<hash>
```

示例：

```text
20260424-201530_train_mitty_three_stage_lora_g1_walk_train_n128_lr1e-4_rauto_a1b2c3d4
```

规则：

- 时间使用本地时区 `Asia/Shanghai`。
- `experiment_type` 必须来自受控枚举。
- `n<samples>` 由数据索引/manifest 自动统计。
- `keyparams` 只包含影响产物语义的短参数，例如 lr、resolution、frames、stage。
- `hash` 基于规范化参数生成，避免超长目录名并保证可追溯。

### 实验记录

每次实验至少写入：

- `experiment_name`、`experiment_type`、创建时间、命令行、工作目录、git commit/status 摘要。
- 数据集根目录、split、样本数、样本清单或索引文件 hash。
- 模型基础版本、LoRA checkpoint、自动检测到的 LoRA rank/dim。
- 关键参数、随机种子、设备、输出路径、cache 路径。
- 生成或训练结果摘要、失败阶段和错误信息（失败时也保留 manifest）。

## 风险与注意事项

- 历史脚本可能仍被旧文档引用；重构时必须同步更新 `doc/scripts_inventory.md`。
- 自动实验命名若包含过多参数会导致目录过长；需用短 hash 控制长度。
- Manifest 写入不能吞掉训练异常；失败应保留已有记录并继续抛出原始异常。
- DiffSynth 解耦可能暴露隐藏 import；需通过 import smoke 和 CLI `--help` 提前发现。
- 大规模数据/cache 不迁移；只对新实验生效，历史实验以 archive 文档说明。
