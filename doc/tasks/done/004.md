## 背景

当前视频微调 `src/pipeline/train_lora.py` 走 Wan 2.1 Fun-V1.1-14B-Control，control_video 通过 FunControl 的 ref_conv 作 pixel-level 条件注入。此方案两个问题：
1. 14B FP8 在 4090 上显存余量 <1GB，任何额外开销都 OOM
2. 若改成 Mitty（2512.Mitty.pdf）的序列拼接方案，Fun-Control 的 conditioning 路径完全不兼容

调研发现 **Wan 2.2 TI2V-5B 架构原生支持 Mitty 所需的 partial-noising 机制**：
- `seperated_timestep=True`：每个时空 patch 独立 timestep
- `fuse_vae_embedding_in_latents=True`：`latents[:,:,0:1]` 被 input_image 的 VAE latent 替换，同时 `timestep[:,:f0]=0` 让第 1 帧保持 clean
- 核心路径见 `diffsynth/pipelines/wan_video.py:1376-1380`

Mitty 方案等价于把这个机制从"1 帧 clean 前缀"扩展到"整段 human 视频 clean 前缀"。换用 TI2V-5B 后**无需改 DiT、无需改 RoPE**，只需扩展 timestep 构造 + noise/loss mask。

本任务聚焦环境搭建和机制验证，不做训练。

## 交付记录

- **权重下载**：`/disk_n/zzf/.cache/huggingface/hub/models--Wan-AI--Wan2.2-TI2V-5B/manual/`
  - `diffusion_pytorch_model-00001..3-of-00003.safetensors` (共 ~20 GB)
  - `models_t5_umt5-xxl-enc-bf16.pth` (10.6 GB)
  - `Wan2.2_VAE.pth` (2.7 GB)
  - `google/umt5-xxl/` tokenizer files
- **Smoke test 脚本**：`scripts/smoke_wan22_ti2v5b.py`
- **测试结果**：
  - 方案 A (全 disk offload): T2V cold 155s / warm 73s, TI2V 23s, peak **10.83 GB**
  - 方案 B (T5 offload + DiT 常驻 + VAE cpu offload): T2V **12.3s**, TI2V **11.8s**, peak **19.88 GB**, 稳态 9.31 GB
  - DiT forward 速度: 3.65 it/s (~270ms/step)
  - 方案 B 比 A 快 6x，对齐 T005 训练配置
- **生成视频**：`output/wan22_smoke/smoke_t2v.mp4`, `smoke_ti2v.mp4`（均 640×480@16fps, 17f, ~87-93 KB，可播放）
- **文档**：`doc/step_5_wan22_ti2v5b.md`（已填实测数据 + Mitty 改造策略）
- **关键发现**：
  - DiffSynth 默认的全 GPU 加载会 OOM（T5+DiT 装满 23GB 后 VAE 进不去）
  - 必须用 low_vram 配置（disk offload + cuda compute），4090 才能跑 5B
  - 训练时同样需要让 T5 offload（data_process 后卸掉），DiT 常驻 GPU
  - seperated_timestep 机制确认工作：`wan_video.py:1376-1380` 构造 `[0]*第1帧 + [t]*其余帧` 的 per-patch timestep，这正是 Mitty 要扩展为 `[0]*前 f_H 帧 + [t]*后 f_R 帧` 的地方
